Place a New Order

This endpoint submits a request to place a new order from user's cash account.

POST api/v2/cash/order

API Key Permission: Trade

Requested Parameters:

FieldName FieldType Required Default Description Value Range
coid string Yes NA a unique identifier of length 32 which consists of only lower case characters (a-z), upper case characters (A-Z) and digits (0-9)
time long Yes NA milliseconds since UNIX epoch in UTC, requests placed more than 30 seconds ago are treated as expired and will not be processed
symbol string Yes NA Trading symbol to trade
orderPrice string No NA required for limit orders and stop orders
stopPrice string No NA required for limit orders and stop orders
orderQty string Yes NA The quantity of the order
orderType string Yes NA the type of the the order "limit", "market", "stop_markter", "stop_limit"
side string Yes NA trader's side "buy", "sell"
postOnly boolean No False if true, the order will either be posted to the limit order book or be cancelled, i.e. the order cannot take liquidity True, False
timeInForce string No "GTC" Currently, we support "GTC" (good-till-canceled) and "IOC" (immediate-or-cancel). "GTC, "IOC"
sync boolean No False if true, the 'status' field in response will demonstrate the status of the order in matching engine (note that it will slow down the response speed) True, False

Response Content:

FieldName FiledType Description
coid string a unique identifier of length 32 (see above)
status string order stauts (see below)

Request body: application/json

Example:

{
    'coid': "98D98gzoVcE1U6UC7C7mupckj4rhOoJ0",
    'time': 1561043031451,
    'symbol': "BTH/ETC",
    'orderPrice': "0.0315",
    'stopPrice': None,
    'orderQty': "1.2",
    'orderType': "limit",
    'side': "buy",
    'postOnly': None,
    'timeInForce': None,
    'sync': None
}

Successful Response Example:

{
   'code': 0,
   'data': {'coid': '98D98gzoVcE1U6UC7C7mupckj4rhOoJ0', 'status': 'submitted'},
   'email': 'kuo.bao@test.com',
   'timestamp': 1561043031959
}

Other Info:

  • For this API you must include x-auth-coid in your request header.
  • Order status: An order may be in one of the following status in the matching engine: pendingNew(only available for stop orders), New, PartiallyFilled, Filled, Canceled, Rejected. If the status is 'submitted', it means the order is valid and has been received by the server. However, it doesn't imply that the order has been successfully submitted to the matching engine. If 'sync' field in request body is false, the 'status' field in response will always be 'submitted'. API users who want to know the status of the order in the matching engine should set 'sync' filed in the request body to be true (note that it will slow down the response speed and the 'status' field in response can still be 'submitted'). API users can always use the list open order API to monitor the status of the order placed.
  • API users should be careful about decimal places for oderPrice, stopPrice and orderQty. The number of decimal places for orderPrice/ stopPrice should not exceed priceScale, and that for orderQty should not exceed qtyScale, otherwise, they will be rounded. (prcieScale / qtyScale for each symbol can be obtained from API GET api/v1/products)
  • The table below shows how to correctly configure order of different types: (o- required, x - optional)
coid time symbol orderPrice orderQty orderType side postOnly stopPrice timeInForce sync
Market o o o o o o x
Limit o o o o o o o x x x
StopMarket o o o o o o o x
StopLimit o o o o o o o o x
